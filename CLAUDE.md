## Architecture

**Two-part system:**

1. **Blog frontend** (`app/(blog)/`) — Next.js SSG reads HTML posts from `/posts/*.html` + metadata from `/posts/*.json`. All post reading goes through `lib/posts.ts` (server-only, uses Node.js `fs`).

2. **PWA Editor** (`app/editor/`) — In-browser Editor.js CMS. Drafts auto-save to IndexedDB via Dexie (`lib/db.ts`). Publishing commits HTML + JSON files to GitHub via Contents API (`lib/github.ts`).

## Critical Constraints

### Tailwind v3 — DO NOT use Tailwind v4 or `@base-ui/react`
All `components/ui/` components are manually written with **Radix UI primitives** + `hsl()` CSS variables. The `shadcn` CLI installs v4-style components incompatible with this project — do not run `npx shadcn@latest add`. Add UI components manually following the existing pattern.

### Editor.js — never import at module level
Editor.js accesses `window`/`document` and breaks SSR. Always import inside `useEffect`:
```ts
const EditorJS = (await import('@editorjs/editorjs')).default
```
`components/Editor/tools.ts` is also imported dynamically. The wrapper component (`components/Editor/EditorWrapper.tsx`) is itself imported with `dynamic(..., { ssr: false })` at all call sites.

### DOMPurify — run only in `useEffect`
`DOMPurify.sanitize()` requires `window` and must not run during SSR render. See `components/PostContent.tsx` for the pattern: sanitize in first `useEffect`, highlight in second.

### GitHub OAuth — token exchange is server-side only
`GITHUB_CLIENT_SECRET` must never be exposed to the client. The OAuth flow goes: client → `/api/auth/github/callback` (Route Handler) → GitHub token endpoint → returns token to client. Token is stored in `sessionStorage` only.

### `lib/posts.ts` — server-only
This file imports `server-only` and uses Node.js `fs`. Never import it in Client Components or any file that can be bundled for the browser.

## Environment Variables

Copy `.env.local.example` to `.env.local`. Required vars:
- `GITHUB_CLIENT_ID` + `GITHUB_CLIENT_SECRET` — GitHub OAuth App (server-side only)
- `NEXT_PUBLIC_GITHUB_CLIENT_ID` — same Client ID, safe for client
- `NEXT_PUBLIC_GITHUB_REPO` — target repo for publishing (e.g. `user/repo`)
- `NEXT_PUBLIC_GITHUB_BRANCH` — branch to commit to (e.g. `main`)
- `NEXT_PUBLIC_GISCUS_REPO`, `NEXT_PUBLIC_GISCUS_REPO_ID`, `NEXT_PUBLIC_GISCUS_CATEGORY`, `NEXT_PUBLIC_GISCUS_CATEGORY_ID` — from giscus.app
- `NEXT_PUBLIC_SITE_URL` — production URL for OG images and sitemap

## Post Format

Posts live in `/posts/` as two files per post:
- `posts/[slug].html` — sanitized HTML content
- `posts/[slug].json` — metadata: `{ title, slug, description, tags, publishedAt, updatedAt }`

The editor publishes both files automatically. To add a post manually, create both files following the `hello-world` seed.

## PWA Notes

Service worker is only generated by `npm run build` (not `dev`). Icons at `public/icon-192.png` and `public/icon-512.png` must be added manually — they are not included in the repo.
